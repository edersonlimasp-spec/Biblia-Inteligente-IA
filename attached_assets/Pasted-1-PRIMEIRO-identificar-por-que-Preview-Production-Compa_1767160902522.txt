1) PRIMEIRO: identificar por que Preview ≠ Production

Compare e registre (em texto) as diferenças reais:
	•	Build/Start command do Preview vs Production
	•	Framework mode (dev vs prod) e flags (ex.: NODE_ENV, NEXT_PUBLIC_*)
	•	Variáveis/Secrets disponíveis no Preview e ausentes na Production
	•	Base URL / domínio (Preview usa *.replit.dev e Production usa *.replit.app)
	•	Banco de dados / storage / buckets / URLs que mudam entre ambientes
	•	Rotas que funcionam no preview e quebram na produção (listar)

Obrigatório: imprimir no console na inicialização (tanto preview quanto prod):
	•	process.env.NODE_ENV
	•	process.env.REPLIT_DEPLOYMENT (ou equivalente)
	•	window.location.origin
	•	“build version” (um hash/versão do app)

⸻

2) GARANTIR QUE O BUILD PUBLICADO É O MESMO DO PREVIEW
	•	Remover qualquer comportamento “dev-only” que esteja habilitado no preview e desabilitado no deploy.
	•	Garantir que a Production usa exatamente:
	•	o mesmo build command
	•	o mesmo output folder
	•	o mesmo start command
	•	Se existe build separado (ex.: Next/Vite/React):
	•	garantir que o deploy está servindo o build gerado, não o código “dev”.

Ação obrigatória:
	•	Ajustar replit.nix / replit.toml / configurações do Deploy para apontar para o mesmo processo e pasta de build do preview.

⸻

3) SINCRONIZAR VARIÁVEIS E SECRETS

O motivo mais comum de diferença é ENV faltando no deploy.
	•	Liste todas variáveis lidas pelo app (ex.: DATABASE_URL, SUPABASE_URL, SUPABASE_ANON_KEY, RESEND_*, MP_ACCESS_TOKEN, etc.)
	•	Verifique quais existem no Preview e quais não existem no Deploy
	•	Copiar todas as Secrets necessárias para o ambiente de Deploy (production)
	•	Se variável for NEXT_PUBLIC_*, garantir que está presente no build (porque Next injeta no build)

⸻

4) CORRIGIR URLS E ORIGINS (PREVIEW vs APP)

Se o app usa redirects/callback (login, pagamento, reset senha):
	•	Atualizar allowed origins / redirect URLs para incluir:
	•	domínio do preview
	•	domínio publicado .replit.app
	•	Se usa cookies/sessão:
	•	configurar cookie domain/sameSite/secure para funcionar em produção também

⸻

5) BANCO / DADOS / SEED — PRODUÇÃO NÃO PODE FICAR “EM BRANCO”

Se em produção “some conteúdo” (ex.: lições), então:
	•	produção está apontando para outro banco, ou
	•	seed não rodou, ou
	•	permissões/paths mudaram.

Obrigatório:
	•	Fazer fallback seguro: se o banco não retornar, carregar conteúdo local (seed/JSON) para não ficar em branco.
	•	Garantir que o mesmo dataset do preview existe na produção.

⸻

6) TRAVA DE VERSÃO (PROVA QUE É O MESMO CÓDIGO)

Implemente um BUILD_ID visível (ex.: no footer oculto ou console):
	•	BUILD_ID = timestamp/commit hash
	•	Deve ser o mesmo no preview e no deploy após publicar.

⸻

7) VALIDAÇÃO FINAL — NÃO CONCLUIR SEM TESTAR

Antes de finalizar, você deve:
	1.	Rodar preview e anotar:
	•	origin
	•	NODE_ENV
	•	BUILD_ID
	•	3 telas críticas (ex.: Home, Cursos Premium, abrir lição)
	2.	Publicar
	3.	Repetir os mesmos testes no domínio publicado
	4.	Confirmar que:
	•	mesma UI
	•	mesmas lições aparecem
	•	abrir lição funciona
	•	login e troca de idioma funcionam
	•	nada “some” em produção

Ao final, me entregue:
	•	causa raiz (o que estava diferente)
	•	arquivos/configurações alteradas
	•	print/log do BUILD_ID e NODE_ENV em preview e production
	•	confirmação: “Production está idêntica ao Preview”
