Quero que você CORRIJA completamente a forma como o Strong está implementado no app “Bíblia Hebraico & Grego – Primeiros Textos + IA”.

O que você FEZ ATÉ AGORA está incorreto conceitualmente:
- Você colocou um mapeamento manual de ~200 palavras dentro do componente (ex.: BibleReader.tsx).
- Pintou em azul apenas essas palavras mapeadas.
- Isso NÃO é aceitável para uma Bíblia com 66 livros e milhares de palavras originais.

Eu NÃO quero mais um mapeamento parcial em memória, eu quero uma INTEGRAÇÃO REAL com o DICIONÁRIO COMPLETO DE STRONG para TODAS AS PALAVRAS.

=====================================================================
1) PARE COM O MAPEAMENTO MANUAL NO COMPONENTE
=====================================================================
- Remova a ideia de ter apenas algumas palavras num objeto hardcoded em BibleReader.
- Nenhum mapeamento de Strong deve ficar “escondido” dentro de componente React.
- Toda a lógica de Strong deve vir de DADOS estruturados (banco ou JSON completo).

=====================================================================
2) USE UM DATASET COMPLETO DE STRONG (HEBRAICO E GREGO)
=====================================================================
Implemente uma abordagem usando um dataset completo, como por exemplo:

- Projeto open-source “openscriptures/strongs” (Hebrew e Greek Strong’s dictionaries em JSON/XML).
- OU outro dicionário Strong completo e público em formato processável.

Passos mínimos:

1. No backend, baixe/clonar o repositório ou traga os arquivos necessários (hebrew + greek Strong) para o projeto.
2. Crie uma tabela no banco, por exemplo:

   Tabela strong_entries:
   - id (PK)
   - strong_number (string, ex: "H7225", "G3056")
   - language (hebrew / greek)
   - original (palavra em hebraico ou grego)
   - transliteration
   - definition (texto em português ou inglês por enquanto)
   - extra_fields (JSON opcional)

3. Crie um script de IMPORTAÇÃO:
   - Ler o JSON/XML do dicionário Strong.
   - Popular a tabela strong_entries com TODAS as entradas (milhares de registros).

=====================================================================
3) VÍNCULO DAS PALAVRAS DA BÍBLIA COM STRONG
=====================================================================
Eu quero que TODAS as palavras da Bíblia (no texto original) tenham um número Strong associado.

Abordagem recomendada:

1. Use um texto bíblico original já marcado com Strong (por exemplo, projetos como o Open Scriptures Hebrew Bible e textos gregos com Strong).
2. Crie uma tabela, por exemplo:

   Tabela bible_words:
   - id
   - book (número/livro)
   - chapter
   - verse
   - word_position (ordem da palavra no versículo)
   - original_word
   - strong_number (ex.: "H7225", "G3056")

3. Ao buscar um versículo, o backend deve retornar NÃO apenas o texto em português, mas também:
   - a lista de palavras com seus respectivos strong_number.

Se você não conseguir alinhar 100% com o português agora, comece com:
- Interlinear ou
- Lista das palavras originais + Strong por versículo.

Mas o importante é:
- Cada palavra exibida como clicável na tela deve ter um strong_number de verdade vindo do banco.

=====================================================================
4) COMPORTAMENTO NO FRONT-END (BIBLEREADER)
=====================================================================
No componente de leitura bíblica:

- Para cada palavra renderizada:
  - Se houver strong_number vinculado:
    • renderizar a palavra em AZUL (ex.: text-blue-600 / dark:text-blue-400),
    • tornar a palavra CLICÁVEL.
  - Se NÃO houver strong_number:
    • renderizar normal (texto padrão).

- Ao clicar numa palavra com Strong:
  1. Fazer chamada ao backend: GET /api/strongs/{strong_number}
  2. Receber:
     - número Strong,
     - palavra original (hebraico/greek),
     - transliteração,
     - definição.
  3. Exibir isso num modal ou painel.

NÃO quero limites do tipo “apenas 200 palavras mapeadas”.
Quero que isso funcione para TODAS as palavras que tenham Strong.

=====================================================================
5) PERFORMANCE E ARQUITETURA
=====================================================================
- Não carregue todo o dicionário Strong no front em memória.
- Use:
  - Banco de dados (PostgreSQL) com índices em strong_number.
  - Paginação / lazy loading conforme o livro/capítulo/versículo.
- Para cada versículo na tela:
  - Buscar apenas as palavras e Strong relacionados àquele versículo.

=====================================================================
6) TESTES E VALIDAÇÃO
=====================================================================
Antes de dizer que terminou:

1. Teste vários textos:
   - Antigo Testamento (hebraico) → palavras com H####.
   - Novo Testamento (grego) → palavras com G####.

2. Verifique que:
   - Ao tocar em QUALQUER palavra do texto original, o Strong aparece.
   - Não há mais limitação de “apenas algumas palavras em azul”.
   - O dicionário Strong está vindo do banco/dataset completo.

3. Me explique no final:
   - Que dataset você usou (nome e origem).
   - Como é o formato dos dados importados.
   - Quais tabelas/estruturas foram criadas.
   - Como a tela BibleReader está recebendo e exibindo os strong_number.

NÃO considere essa tarefa concluída enquanto:
- ainda existirem poucas palavras com Strong,
- ou enquanto o dicionário estiver “hardcoded” no código,
- ou enquanto o mapeamento não cobrir todo o texto original de forma consistente.
