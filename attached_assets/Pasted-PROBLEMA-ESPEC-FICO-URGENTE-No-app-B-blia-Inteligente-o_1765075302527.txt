PROBLEMA ESPECÍFICO (URGENTE):

No app “Bíblia Inteligente” o STRONG funciona normalmente no PREVIEW (modo desenvolvimento), mas PARA DE FUNCIONAR depois que o app é PUBLICADO.

Quero uma correção DEFINITIVA para que o Strong funcione IGUAL em:
- Preview (dev)
- App publicado (produção – URL pública)

NÃO quero tentativas genéricas. Quero que você:
1) Reproduza o erro na versão publicada.
2) Descubra a causa EXATA.
3) Corrija.
4) Teste novamente NA URL PÚBLICA.

====================================================
1) REPRODUZIR O ERRO NA VERSÃO PUBLICADA
====================================================
1. Abra a URL PÚBLICA do app publicado.
2. Vá até a tela de leitura da Bíblia.
3. Clique em palavras que possuem Strong (hebraico/grego).

Enquanto faz isso:
- Abra o DevTools do navegador (F12).
- Na aba Console:
  → veja se aparece algum erro ao clicar.
- Na aba Network:
  → veja se alguma requisição é feita (para /strong, /api/strong, etc.)
  → veja o status (200, 404, 500…).

Você DEVE anotar:
- qual mensagem de erro aparece no CONSOLE (se houver);
- qual endpoint é chamado (ou não é chamado);
- qual status a resposta retorna.

====================================================
2) VERIFICAR DADOS DO STRONG EM PRODUÇÃO
====================================================
Verifique no banco de dados de PRODUÇÃO (não apenas o de desenvolvimento):

- se a tabela/coleção do Strong está realmente preenchida (contar registros).
  Ex.: SELECT COUNT(*) FROM strong_entries;

Se em produção não houver entradas:
- precisa rodar o SEED para popular o Strong também no banco de produção.
- garantir que a estrutura de deploy (build + publish) NÃO usa um banco vazio.

Se em produção o Strong estiver cheio (ex.: 14.000+ entradas):
- então o problema está no backend (rota) ou no frontend (chamada).

====================================================
3) VERIFICAR ROTA / API DO STRONG EM PRODUÇÃO
====================================================
Identifique qual endpoint o frontend usa (exemplos):

- /api/strong/:id
- /strong/:id
- algo semelhante.

No ambiente PUBLICADO, teste esse endpoint diretamente:
- pelo browser
- ou via curl na URL pública.

Se der 404:
- corrigir o arquivo de rotas do backend (Express/Nest)
- garantir que essa rota está registrada ANTES do fallback de SPA (app.get("*", ...)).

Se der 500:
- olhar o log do servidor em produção e corrigir erro de código.

IMPORTANTE:
Verificar se em produção o frontend não está chamando http://localhost:XXXX.
Se isso estiver ocorrendo:
- substituir por caminho RELATIVO: "/api/strong/:id" (sem localhost).

====================================================
4) VERIFICAR FRONTEND – CLIQUE E STRONGID
====================================================
No código que renderiza o texto bíblico:

- conferir se, no build de produção, cada palavra ainda vem com strongId.
- inspecionar o DOM em produção numa palavra que deveria ter Strong:
  → ver data-attributes ou props que contém o strongId.

Se o strongId NÃO estiver chegando:
- ver se existe alguma lógica diferente para NODE_ENV=production.
- ver se há alguma função que “limpa” campos no build de produção.

Se o strongId estiver ok, mas o clique não faz nada:
- verificar se o onClick está realmente atribuído ao elemento.
- garantir que a função que abre o modal Strong NÃO está sendo removida/“tree shaken” no bundle de produção.

====================================================
5) AJUSTAR BUILD / CAMINHO DE ARQUIVOS (SE STRONG FOR JSON/ARQUIVO ESTÁTICO)
====================================================
Se os dados do Strong estiverem em arquivos JSON ou estáticos:

- conferir se esses arquivos estão sendo copiados corretamente para a pasta que o servidor serve (ex.: server/public, dist, etc.).
- se necessário, ajustar o comando de build/publicação para que esses arquivos sejam copiados para a pasta correta em produção.

Exemplo: (AJUSTAR SE FOR O CASO)
npm ci && npm run build && mkdir -p server/public && rm -rf server/public/* && cp -r dist/* server/public/

====================================================
6) TESTES FINAIS – APENAS EM PRODUÇÃO
====================================================
Depois de corrigir, OBRIGATORIAMENTE:

1. Publicar novamente o app.
2. Acessar a URL pública em:
   - Navegador normal
   - Modo anônimo
   - Celular (se possível)
3. Abrir um texto do Antigo Testamento e clicar em 3–4 palavras com Strong.
4. Abrir um texto do Novo Testamento e clicar em 3–4 palavras com Strong.

O comportamento correto deve ser:
✔ Ao clicar em palavras com Strong:
   - abrir modal/caixa com:
     • número Strong (Hxxxx ou Gxxxx),
     • palavra original,
     • transliteração,
     • definição.

✔ NÃO devem aparecer erros no console.
✔ A chamada à API deve responder 200 com os dados do Strong.
