ETAPA 2 – ARQUITETURA CORRETA (OBRIGATÓRIA)
----------------------------------------------------

1) ENDPOINTS EM 2 CAMADAS (OBRIGATÓRIO)
- /api/strong/summary/:id
  -> resposta < 20kb
  -> strongNumber, lemma, transliteration, shortDefinition
- /api/strong/detail/:id
  -> textos longos, referências, notas

O modal abre com summary INSTANTÂNEO.
Detail carrega em background.

2) CACHE LOCAL-FIRST UNIVERSAL (CRÍTICO)
Implementar:
- IndexedDB (idb)
- Cache key: strongNumber
- Fluxo:
  a) procura em IndexedDB → se existir, abre imediatamente
  b) se não existir, busca summary → salva → abre
  c) busca detail em background → atualiza

⚠️ NÃO confiar apenas em Service Worker (iOS é instável).

3) SERVICE WORKER (APENAS COMO APOIO)
- Estratégia: stale-while-revalidate para /api/strong/*
- Cache versionado
- Fallback automático para IndexedDB se SW falhar

4) PRÉ-CARREGAMENTO INTELIGENTE
Ao abrir um capítulo:
- identificar todos os strongNumbers do capítulo
- pré-carregar em background os primeiros 30 summaries
- salvar em IndexedDB

Resultado esperado:
→ 90% dos cliques instantâneos

----------------------------------------------------
ETAPA 3 – BACKEND EXTREMAMENTE RÁPIDO
----------------------------------------------------

1) BANCO
- Garantir índice:
  - strongNumber (UNIQUE)
- Query SEM LIKE, SEM JSON pesado

2) CACHE NO SERVIDOR
- LRU cache por strongNumber
- TTL >= 24h
- HIT deve responder < 50ms

3) COMPRESSÃO
- Ativar gzip/br no Express
- Confirmar que produção está comprimindo
